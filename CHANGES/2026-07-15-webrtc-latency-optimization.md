# WebRTC Latency Optimization Session Summary

## 1. Initial Goal
Implement Milestones 1 to 5 to optimize WebRTC connection setup latency and audio-video playout latency for TV and mobile devices. This includes:
- Exposing the `isTvDevice()` detection utility in the Android native WebView bridge.
- Pre-fetching TURN credentials upon Firebase authentication.
- Implementing ICE candidate queueing and batching to minimize Firestore writes.
- Adjusting SDP configuration to support low-latency Opus audio parameters (`ptime=10` and `minptime=10`).
- Dynamically tuning media constraints (video resolution/frame rate) for Android TV devices vs. mobile devices.
- Setting `playoutDelayHint = 0` on incoming track receivers for minimal jitter buffering latency.
- Verifying the build on both React (Web) and Android environments.

## 2. Design Choices Considered
- **ICE Candidate Write Strategy**: Considered writing candidate documents to Firestore individually as they are generated by the peer connection. However, WebRTC candidates are generated rapidly, causing high firestore write traffic and connection setup delay. We decided to queue candidates locally and batch-commit them using Firestore's `writeBatch` either after a 500ms timeout or when the ICE candidate generation completes.
- **Opus Latency Settings**: Considered editing only the `useinbandfec=1` attribute in the Opus format configuration line. However, to guarantee 10ms frame packing and a low max average bitrate (20kbps), we decided to explicitly inject `ptime=10;minptime=10;maxaveragebitrate=20000` into the format configuration line and insert a standalone `a=ptime:10` attribute directly below the audio media section descriptor (`m=audio`).
- **TV vs Mobile Constraints**: Considered using the same 640x480 @ 24fps constraints for both mobile and TV. However, TVs have lower hardware acceleration and decoding power in webview contexts. So we introduced a dynamic helper that scales TV down to 640x360 @ 15fps (max 960x540 @ 20fps) to drastically reduce encoder/decoder overhead, while keeping 640x480 @ 24fps for mobile clients.
- **TURN Credential Fetching**: Flowing credentials only on-demand inside `setupWebRTC` can cause a delay in ICE candidate gathering. We opted to pre-fetch credentials on login, store them in React state, and dynamically reconfigure the `RTCPeerConnection` if they arrive late, falling back to on-demand fetching and STUN servers if needed.

## 3. Design Choices Chosen
- **Exposed `isTvDevice()` in MainActivity**: Used `JavascriptInterface` to make the TV detection method available to the web client.
- **Pre-fetched TURN credentials**: Triggered TURN credential fetching upon successful authentication, reducing connection startup time.
- **ICE Candidate Batching**: Committed queued candidates using a `writeBatch` in Firestore to reduce transaction overhead.
- **Low Latency Opus parameters**: Modified SDP formatting in `adjustSdp` to force Opus mono and inject `ptime=10`, `minptime=10`, `maxaveragebitrate=20000`, and `a=ptime:10`.
- **TV Media Constraints**: Configured 640x360 @ 15fps for TV devices to prevent decoding lag, and 640x480 @ 24fps for standard devices.
- **playoutDelayHint**: Configured `playoutDelayHint = 0` on incoming RTCRtpReceiver tracks to request the browser to minimize receiver buffer latency.
- **Version Compatibility**: Verified that these changes are backwards compatible with older clients and do not require incrementing `minClientVersion`.
